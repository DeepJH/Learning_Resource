📌 Git Merge 做了什么？

✨ 关键点: Merge 会创建一个新的合并提交 M，它有两个父提交（C 和 E），完整保留了两个分支的历史。

# 步骤 1: 在 main 分支上
git checkout main

# 当前状态: HEAD 指向 main 的最新提交 C
# 步骤 2: 执行合并命令
git merge feature

# Git 会找到两个分支的共同祖先（A）
# 然后计算 C 和 E 的差异
# 步骤 3: Git 自动合并（如果没有冲突）
# - 合并两个分支的所有文件变更
# - 创建新的合并提交 M，有两个父提交

# 如果有冲突，Git 会暂停让你手动解决
# 步骤 4: 合并完成！
# main 现在包含了 feature 的所有更改
# 历史完整保留: A→B→C→M 和 A→D→E 都存在

git log --graph # 可以看到合并提交 M 有两条父线

🔄 Git Rebase 做了什么？

# 步骤 1: 切换到 feature 分支
git checkout feature

# 当前 feature 基于 A 分叉出去，有 D、E 两个提交
# 步骤 2: 执行 rebase
git rebase main

# Git 会找到共同祖先 A
# 然后把 feature 的提交"摘下来"暂存
# 步骤 3: Git 先将 feature 的改动暂存
# - 保存 D 的变更为补丁 patch-D
# - 保存 E 的变更为补丁 patch-E
# - 将 feature 分支指向 main 的最新提交 C

# 现在 feature 指向 C，补丁 D、E 待应用
# 步骤 4: 逐个应用补丁
# Git 在 C 的基础上应用 patch-D → 创建新提交 D'
# Git 在 D' 的基础上应用 patch-E → 创建新提交 E'

# 如果有冲突，会停下来让你解决
# 解决后: git add . && git rebase --continue
# 步骤 5: Rebase 完成！
# feature 现在是: A→B→C→D'→E'（一条直线）
# 旧的 D、E 被丢弃（灰色显示）

git push -f origin feature # 需要强制推送

⚖️ Merge vs Rebase 对比

📌 Merge

  • ✅ 安全，不修改历史
  • ✅ 适合共享分支
  • ❌ 历史会有分叉
  • ❌ 多次 merge 会杂乱

  • ✅ 便于理解变更顺序

🎯 黄金法则: 永远不要 rebase 已经被 push 到远程、别人可能已经拉取的分支！ 只 rebase 你自己的本地分支。

🔀 PR 的三种合并方式

在 GitHub/GitLab 上合并 PR 时，通常有三种选择：

1️⃣ Create a Merge Commit（创建合并提交）

相当于 git merge --no-ff，保留完整的分叉历史和 PR 信息。

# 结果: main: A → B → C → M ↗ feat: D → E ↗

适用场景: 需要保留功能分支的完整开发历史。

2️⃣ Squash and Merge（压缩并合并）

将 PR 的所有提交压缩成一个提交，然后合并到 main。历史最简洁。

# 结果: main: A → B → C → [D+E] # feature 分支的 D 和 E 被压缩成一个提交

适用场景: 功能分支有很多小提交（"fix", "wip"），只想让 main 保持整洁。

3️⃣ Rebase and Merge（变基并合并）

将 PR 的每个提交逐个 rebase 到 main 上，不压缩，保留每个提交。

# 结果: main: A → B → C → D' → E' # 每个提交都被重新基于 main 创建

适用场景: 想保留每个提交的独立性，又想要线性历史。

# 1️⃣ Merge Commit 方式

git checkout main
git merge --no-ff feature -m "Merge PR #1"

# 结果: A→B→C→M
# ↗
# D→E↗
# ✅ 保留完整历史，包括 PR 信息
# 2️⃣ Squash and Merge 方式

git checkout main
git merge --squash feature
git commit -m "Feature: 添加了XX功能"

# 结果: A→B→C→[D+E]
# ✅ main 历史整洁，一个功能一个提交
# ❌ 丢失了 feature 分支的提交细节
# 3️⃣ Rebase and Merge 方式

# GitHub 内部执行类似:
git checkout feature
git rebase main
git checkout main
git merge feature # 此时是快进合并

# 结果: A→B→C→D'→E'
# ✅ 保留每个提交的独立性，线性历史

⚠️ 多人协作：PR 如何解决冲突

当多个人修改了同一个文件的同一部分，就会产生冲突。

😱 冲突产生

# main 上 Alice 改了: def hello(): print("Hello Alice") # PR 上 Bob 也改了: def hello(): print("Hello Bob")

❌ Git 不知道该保留谁

🔧 解决冲突

# 1. 拉取 main 的最新 git checkout feature git pull origin main # 2. Git 标记冲突 # 3. 手动编辑冲突文件 # 4. 标记已解决 git add . git commit

✅ 冲突解决

# 解决后的代码: def hello(name): print(f"Hello {name}") # PR 自动更新！ # GitHub 会重新检查 CI # 审查者可以看到解决后的 diff

✓ PR 可以继续合并

💡 最佳实践: 在 PR 期间，定期将 main 的变更合并到 feature 分支，可以提前发现冲突，避免最后一次性解决大量冲突。

git add config.py
git commit -m "Merge main: resolve config conflict"

🔧 写一半 Main 更新了怎么办？

这是日常开发最常见的场景！你在 feature 分支开发到一半，main 已经有了新的提交。

# 你的场景: main: A → B → C → D → E # main 有了新提交 D, E feature: A → B → F → G # 你还在 F, G 开发中

你有两种选择来同步 main 的变更：

方案 1: Merge

git checkout feature
git merge main

优点:

  • ✅ 简单安全
  • ✅ 保留历史记录
  • ✅ 容易回滚

结果:

feature: A→B→F→G→M
                ↗
main: A→B→C→D→E↗

方案 2: Rebase

git checkout feature
git rebase main

优点:

  • ✅ 历史线性整洁
  • ✅ 便于后续 cherry-pick
  • ❌ 会重写你的提交

结果:

feature: A→B→C→D→E→F'→G'

# 现状分析:
# - 你在 feature 分支开发了 F, G
# - main 已经有了新提交 C, D, E
# - 你的 feature 是基于旧 main (B) 的

# ⚠️ 需要同步 main 的变更，避免冲突积累
# 方案 1: 使用 Merge 同步

git checkout feature
git fetch origin
git merge origin/main

# Git 会:
# 1. 找到共同祖先 (B)
# 2. 合并 C, D, E 到 feature
# 3. 如果有冲突，停下来让你解决
# 4. 创建合并提交 M
# 方案 2: 使用 Rebase 同步

git checkout feature
git fetch origin
git rebase origin/main

# Git 会:
# 1. 将 feature 指向 main 最新的 E
# 2. 逐个重放 F, G 到新的基础上
# 3. 如果某个提交有冲突，暂停
# 4. 你解决后: git add . && git rebase --continue
# 🎯 应该选哪个？

# ✅ 推荐: 本地开发用 Rebase，共享分支用 Merge
# - 只有你用 feature → rebase (历史整洁)
# - 别人也基于 feature → merge (不改写历史)

# 📝 最佳实践:
# 定期同步 main，不要等到 PR 前才合并！
# 小步快跑，冲突少且容易解决

🎯 总结与最佳实践

1. Merge vs Rebase
• Merge 保留完整历史，适合共享分支
• Rebase 让历史线性，适合本地整理
• 永远不要 rebase 公共分支！
2. PR 合并方式选择
• Merge Commit: 需要保留完整开发历史
• Squash: 功能分支有很多小提交，想保持 main 整洁
• Rebase: 想保留每个提交独立性，又要线性历史
3. 冲突解决
• 定期同步 main 到 feature，提前发现冲突
• 冲突不可怕，Git 会标记冲突位置，手动解决即可
• 解决后推送，PR 自动更新
4. Main 更新了怎么办？
• 只有你用 feature 分支 → git rebase main
• 别人也基于 feature → git merge main
• 小步快跑，频繁同步，冲突少且好解决
🚀 掌握了这些，你就能自信地处理任何 Git 场景了！
